# 旧接口与旧代码下线计划

## 目的

本文件用于给“病种/DIP 旧主线”做分阶段退役说明，避免在当前运营驾驶舱已上线后，后续维护人员误把旧接口和旧页面继续接入新功能。

## 当前状态

已完成：

- 前台主路由已切换为：
  - `运营驾驶舱`
  - `科室分析`
  - `报表中心`
  - `数据导入中心`
- 以下后端接口已在 OpenAPI 中标记为 `deprecated`：
  - `/api/v1/dip/*`
  - `/api/v1/director/topic*`
  - `/api/v1/analytics/cost-structure`
  - `/api/v1/analytics/cost-trend`
  - `/api/v1/analytics/clinical-top`
  - `/api/v1/analytics/disease-priority`
- 以下前台旧页面已删除：
  - `DipStatsPage.tsx`
  - `DiseasePriorityPage.tsx`
  - `VulnerabilityPage.tsx`
  - `WorkOrderPage.tsx`
  - `RecordsPage.tsx`
- 以下前端旧 API 包装已删除：
  - 病种优先级分析相关请求
  - 漏洞规则 / 工单相关请求
  - DIP 前台统计与映射相关请求
  - 旧 `director/topic` 前端请求
  - 明细查询 `records/*` 前端请求

## 下一阶段建议

### Phase 1. 停止新接入

执行标准：

- 新页面与新报表只允许使用 `/operations/*`
- 不再为 `dip/*`、`director/topic*`、旧 `analytics/*` 添加任何新字段或 UI 能力
- 如必须兼容历史脚本，只允许修复阻断性 bug，不扩展功能

### Phase 2. 清理未接线路径

当前已完成：

- 前端未接线路径与孤立 API 包装已完成清理

当前可优先复核的对象：

- 是否仍存在外部脚本、运维工具或临时流程依赖旧后端接口
- 前端 `api.ts` 中仍保留但未进入当前主线的历史聚合请求：
  - `fetchCostStructure`
  - `fetchCostTrend`
  - `fetchClinicalTop`
  - `fetchMonthlyReport`
  - `fetchExecutiveReport`

说明：

- 当前剩余对象不一定要立刻删除，但应先确认是否仍被外部脚本、测试或临时管理流程依赖
- 若确认无外部依赖，可在下一轮继续收缩前端聚合 API 或直接下线后端旧接口

### Phase 3. 下线旧后端接口

建议顺序：

1. 先删除前端中所有对旧接口的引用与类型定义
2. 再删除旧 routers 中对应端点
3. 最后清理旧 service / schema 中仅服务旧主线的结构

建议删除顺序：

1. `backend/app/routers/analytics.py` 中旧运营前端不再使用的 disease-focused 端点
2. `backend/app/routers/director.py`
3. `backend/app/routers/dip.py`

## 删除前检查

每次实际删除前，请至少确认以下事项：

- 前端构建通过
- 后端定向测试通过
- 没有运维脚本、导出脚本或外部调用仍依赖旧端点
- README / UAT / 发布说明已同步更新

## 当前推荐策略

当前仓库最稳妥的策略仍然是：

- 短期：保留旧后端接口但不再扩展
- 中期：清掉未接入页面和前端旧 API 包装
- 后期：统一删除旧后端 routers / services / schemas

这样可以最大限度降低运营驾驶舱刚上线阶段的回归风险。
