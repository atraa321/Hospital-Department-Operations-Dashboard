## 2. 约束与前提

- 部署环境：医院内网，不依赖公网服务。
- 后端服务器：Windows Server（建议 2019/2022）。
- 浏览器终端：Chrome/Edge 内网终端。
- 数据来源：Excel 批量导入为主（病案、费用明细、费用汇总、DIP/ICD映射）。
- 安全要求：RBAC、审计日志、脱敏导出、HTTPS（内网证书）。

## 5. 内网部署方案（Windows Server）

### 5.1 拓扑（推荐）

- `WS-APP-01`：IIS + Frontend静态资源 + FastAPI服务。
- `WS-DB-01`：医院现网 MySQL 数据库（可与APP同机，建议后续分离）。
- `FS-01`：导入文件与报表导出目录（内网共享）。

### 5.2 端口规划

| 服务 | 端口 | 说明 |
|---|---|---|
| IIS HTTPS | 443 | 前端与API统一入口 |
| FastAPI | 18080（内网） | 仅IIS可访问 |
| MySQL | 3306（内网白名单） | 仅应用服务器访问 |

### 5.3 部署步骤（摘要）

1. 安装 Python 3.12、IIS、URL Rewrite、ARR（MySQL使用医院现网已部署实例）。
2. 在现网MySQL创建业务库与账号，执行 Alembic 迁移。
3. 构建前端静态文件并发布到 IIS 站点目录。
4. 后端通过 `nssm` 注册为 Windows 服务（指向 `uvicorn` 启动命令）。
5. IIS 配置 `/api` 反向代理到 `127.0.0.1:18080`。
6. 配置 HTTPS 内网证书、日志目录、备份任务与监控脚本。

## 6. 前端设计规范（按 `skills/frontend-design`）

### 6.1 设计原则

- 保持“数据密集但可读”的医疗运营风格，避免通用模板化后台外观。
- 强调信息优先级：关键指标 > 异常预警 > 明细下钻。
- 使用统一变量体系（颜色、间距、字号、阴影、圆角、动效时长）。

### 6.2 明确禁止

- 禁用默认审美组合：Inter/Arial + 白底紫渐变 + 标准卡片栅格套路。
- 禁用无意义动画和过度装饰。

### 6.3 建议视觉方向（本项目）

- 风格：医疗运营“杂志编辑风 + 工业信息图风”。
- 字体：
  - 标题：`DIN Condensed`（本地部署字体文件）
  - 正文：`Source Han Sans SC` 或 `Noto Sans SC`（本地托管）
- 主色：深青蓝 + 暖灰底，强调色使用橙红/湖蓝用于告警分层。
- 动效：仅在页面进入、图表加载、预警状态切换时使用。

### 6.4 工程要求

- 组件分层：`layout` / `domain` / `shared`。
- 图表配置独立为 `chart-presets`，禁止散落硬编码。
- 所有样式需走 Design Token，不允许页面私有魔法值泛滥。

## 7. 安全与合规基线

- 认证：用户名密码 + 会话超时（30分钟）。
- 授权：RBAC + 数据域（全院/本科室/授权科室）。
- 脱敏：姓名、证件号、手机号导出默认脱敏；管理员可审计解密查看。
- 审计：导入、导出、规则变更、工单关闭必须记录操作者与时间。
- 防护：SQL注入、XSS、CSRF 基础防护；上传文件类型与大小限制。
- 备份：数据库每日备份 + 导入原始文件保留90天（可配置）。

## 10. 风险与备选

| 风险 | 说明 | 备选方案 |
|---|---|---|
| Redis在Windows维护成本高 | 官方Windows支持有限 | 一期不引入Redis，改为DB+定时任务 |
| 字体外网不可拉取 | 内网环境常见 | 字体文件本地化托管 |
| 大文件导入占用内存 | 费用明细行数大 | 分块读取 + 批次写入 + 异步任务 |
| 医院策略限制安装Nginx | 运维策略差异 | 采用IIS反向代理统一入口 |

## 11. 最终结论

项目技术栈确认如下：

- 前端：`React + TypeScript + Vite + ECharts + Framer Motion + SCSS`
- 后端：`Python 3.12 + FastAPI + SQLAlchemy + PyMySQL + pandas/openpyxl`
- 数据库：`MySQL 8.0+（医院现网）`
- 部署：`Windows Server + IIS反向代理 + Uvicorn(NSSM托管)`

该方案满足：

- 内网可部署、可审计、可维护；
- Excel数据导入与病种分析场景的开发效率；
- 后续二期扩展（规则中心、闭环治理、来源转化评估）的技术连续性。
